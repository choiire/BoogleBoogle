🎮 Project Overview

BuggleBuggle is a 2D platform arcade game developed in C using Allegro5. Players shoot bubbles to capture enemies and clear 3 stages to achieve the highest score.

✨ Key Features

  • Player Controls: Arrow keys to move, Up to jump, Space to shoot bubbles
  • Enemy AI: Basic monster (ground movement), Throw monster (flying + projectile attack)
  • Bubble Mechanic: 3-stage lifecycle: Shoot → Float → Pop
  • Score System: SQLite3-based leaderboard saving and retrieval
  • 3 Stages: Progressive difficulty (⭐ ~ ⭐⭐⭐)
  • Collision Detection: Precise physics based on AABB algorithm

🎮 Game Manager (game_manager.c)

The central hub of the project, managing all game states and resources. Without the Game Manager, not a single function can operate correctly.

🔑 Key Functions

  • GAME_MANAGER_UpdateState(): Updates and returns the game state
  • GAME_MANAGER_SetGameState(): Transitions to a new game state
  • GAME_MANAGER_CheckCollision(): Resolves collision between all objects
  • GAME_MANAGER_UpdatePhysics(): Applies physics to all objects
  • GAME_MANAGER_UpdateObject(): Updates the state of all objects

🎮 Gameplay Mechanics

BuggleBuggle creates unique gameplay through its bubble-based enemy capture mechanic.

🎯 Basic Game Flow

1. Stage Load └─ Player spawn: (15, 210) └─ Enemy placement: initialized from map data 2. In-Game (INGAME) └─ Each frame: ├─ Input processing (keyboard) ├─ Collision Detection ├─ Physics application ├─ State update ├─ Rendering └─ Stage check 3. Win Condition └─ All enemies defeated → Next stage 4. Lose Condition └─ Lives = 0 → NAME screen (enter name) 5. Final Screen └─ SCORE screen (Leaderboard)

💧 Bubble Mechanic

  • Shoot: SPACE fires forward from player (speed: 3.0 px/f)
  • State 1 - SHOOTING (0~30 frames): Horizontal linear movement
  • State 2 - FLOAT (30~300 frames): Slowly rises (speed: -0.8 px/f)
  • State 3 - POP (300~330 frames): Pop animation
  • Collision: Wall hit during SHOOTING → transitions to FLOAT
  • Enemy-Bubble Interaction: SHOOTING bubble hits enemy → enemy enters TRAPPED state
💡 Design Intent: Bubbles are not mere projectiles — they are floating traps. Players can use bubbles to capture enemies on high platforms or clear their own path.

👾 Enemy AI State Machine

┌─────────────┐ │ IDLE (Start) │ └──────┬──────┘ │ ▼ ┌─────────────┐ │ MOVE │ ← Default state │ - Move L/R │ (auto-proceed) │ - Try jump │ └──────┬──────┘ │ Bubble hit ▼ ┌──────────────┐ │ TRAPPED │ ← Captured state │ (timer) │ (300 frames) │ - Countdown │ └──────┬──────┘ │ Timer ends ▼ ┌──────────────┐ │ MOVE (angry) │ ← is_angry = true │ - 2x speed │ │ - Boosted ATK │ └──────────────┘

🏃 Movement by Enemy Type

  • BASIC: Left/right on ground, attempts to jump over height differences (50% chance)
  • THROW: Flying, reverses direction on wall hit, fires projectile at player every 180 frames

🎯 Score Calculation System

  • Basic Monster Kill: +100 pts
  • Throw Monster Kill: +200 pts
  • Remaining Lives (on stage clear): +500 pts × lives remaining
  • Early Clear Bonus: None (difficulty increase is the reward)

Example Calculation

Stage 1: 6 BASIC kills = 6 × 100 = 600 pts Stage 2: 3 THROW kills = 3 × 200 = 600 pts Stage 3: 3 THROW + 3 BASIC = (3×200) + (3×100) = 900 pts Remaining Lives: 2 × 500 = 1000 pts Total = 600 + 600 + 900 + 1000 = 3100 pts

📐 Map Structure

  • Map Size: 36×24 tiles (864 tiles total)
  • Tile Size: 10×10 pixels
  • Screen Resolution: 320×240 px (buffer) → 960×720 px (3x scaled)
  • Platforms: In map data, 1 = tile, 0 = air
  • Map Boundaries: Automatic boundary handling at top (y=10) and bottom (y=220)

Tile Placement Principles

  • Stage 1: Regular platform layout (for movement practice)
  • Stage 2: Narrower gaps and height differences (forces dodging)
  • Stage 3: Complex maze-like structure (highest difficulty)
⚠️ Warning: If an enemy's initial position is inside a tile, it will immediately be ejected. Always set spawn positions in air (0) cells.
